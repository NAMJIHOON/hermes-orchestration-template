# /triage — 업무 요청 자동 분류 및 실행

요청 내용: **$ARGUMENTS**

## 분류 기준

아래 기준으로 레벨을 결정한다. 복잡도가 높은 쪽으로 올림(round-up)한다.

| 신호 | Task | Sprint | Project |
|------|------|--------|---------|
| 필요한 스킬 수 | 1개 | 2~4개 | 5개 이상 또는 4페이즈 전부 |
| 예상 소요 시간 | ~30분 | ~반나절 | 수일 이상 |
| 산출물 | 파일/PR 1개 | 여러 파일 묶음 | 전체 릴리스 |
| 승인 필요 여부 | 불필요 | 1회 | 페이즈마다 |

## 키워드 빠른 매핑

- **Task 신호어**: 고쳐줘, 작성해줘, 추가해줘, 리뷰해줘, 문서화, 단일 기능, 버그
- **Sprint 신호어**: 이번 주 백로그, N개 처리, 기능 세트, 개선안 묶음
- **Project 신호어**: 새로 만들어, 서비스 구축, 마이그레이션, 런칭, 전체 리팩터링

## 실행 절차

1. 요청 분석 후 레벨을 한 문장으로 판단한다:
   > "이 요청은 [Task/Sprint/Project]입니다. 이유: [한 문장]"

2. **Task**이면 → 즉시 실행
   ```bash
   claude -p "/task $ARGUMENTS"
   ```

3. **Sprint**이면 → 범위 확인 후 실행
   ```
   다음 태스크들로 분해됩니다:
   - [태스크 1] → [스킬]
   - [태스크 2] → [스킬]
   진행할까요? (yes / 수정)
   ```
   승인 받으면:
   ```bash
   hermes run hermes/workflows/sprint.yaml --tasks "$TASK_LIST"
   ```

4. **Project**이면 → 킥오프로 전달
   ```
   전체 프로젝트 파이프라인이 필요합니다.
   artifacts/A-pm/goal.md에 목표를 저장하고 /kickoff를 실행하겠습니다.
   진행할까요? (yes / 수정)
   ```

## 분류 불확실 시

스킬 수가 경계에 걸리면 Task 쪽으로 내림(round-down)하고,
실행 중 범위가 커지면 사용자에게 알리고 레벨 업그레이드를 제안한다.
